multi-agent 병렬 오케스트레이션이 실전 규모로

nested subagents로 context 관리

Boris Cherny가 Claude Code에 depth=5로 제한된 nested subagent(에이전트가 에이전트를 재귀 생성)를 출시. context를 더 잘 관리하는 메커니즘으로 실험 중.

long-running autonomy + self-verification loop

Opus/Sonnet 5를 시간·일 단위로 자율 실행하는 것이 실전 가능. self-verification loop가 모델을 오래 돌리면서도 결과를 사용자 의도에 가깝게 유지하는 핵심 재료.

'loops / software factory' 논쟁과 harness-as-product

AIEWF에서 autonomous coding loop의 실현 가능성 논쟁. Introspection의 autoresearch는 agent가 eval·human feedback으로 시스템을 유지·개선하는 outer loop를 제안. harness가 곧 제품이라는 테제 부상.

Claude Code auto mode 워크플로우

Boris Cherny의 #1 팁은 auto mode — permission prompt 제거로 multi-clauding(병렬 세션)의 기본 building block. /goal, /loop로 완료까지 지속 실행, 클라우드 실행으로 랩톱을 닫아도 계속 작업.

🧪 기회 — 무엇을 만들까

  • 소규모 자립 모듈(~2-5k 줄)에 conformance test를 먼저 짜고 git worktree로 3-5개 parallel Claude Code subagent를 띄워 다른 언어로 재구현 — 실패 시 코드가 아니라 생성 프롬프트를 고치는 Bun 패턴을 소규모로 검증.
  • 'security sweep' harness 구축: Opus/Sonnet 5로 ~5-10개 parallel agent가 디렉터리 슬라이스별 스캔→sqlite-utils 공유 DB에 findings 기록→consolidation agent가 dedupe·랭킹. 시간당 스캔 줄 수와 false-positive율 측정.
  • tool-schema 충실도 A/B: Sonnet 5(v2.1.197)와 구형 모델에 동일 custom edit tool을 주고 20개 동일 작업 실행, 거부·hallucinate된 필드를 로깅해 'better model, worse tool' 회귀를 정량화.
  • minimal 'loop' harness: 고정 spec+eval을 통과할 때까지 Claude Code를 재시작(N회 캡)하며 정체·과편집 지점 로깅. one-shot vs 5-loop vs 15-loop 성공률 비교로 loop 수렴 여부 검증.
  • LangSmith 스타일 online eval + trace 기반 diagnose-and-improve loop를 얹은 agent observability — 프로덕션 trace에서 반복 실패 패턴을 저비용 fine-tuned judge로 탐지.

⚠️ 리스크 — 무엇을 경계할까

  • multi-agent 실패 모드: 경쟁 벤더의 AI code-review agent가 끝없는 disagreement loop에 빠져 수백 개 코멘트와 수만 달러 inference 비용을 유발. 고정 자동 agent 개수를 무비판적으로 쓰지 말 것.
  • cognitive debt: agent가 크고 정교한 변경을 만들수록 개발자 이해가 실제 코드와 괴리. 프로덕션 코드 리뷰를 멈추거나 unreviewable PR을 만들지 말 것.
  • runaway 토큰 비용: 한 명의 개발자가 coding agent로 주간 수천 달러를 소진할 수 있음 — tokenomics/usage 관측이 필수.
  • supply chain: unpinned dependency 자동 최신 해석으로 인한 대규모 감염(예: axios/litellm 자격증명 유출). agent 실험 tooling조차 위험, release-age·container·pinning 방어 필요.
  • 벤치마크 정치·프로바이더 종속: 벤치마크는 gaming 대상이며, agentic 워크로드가 특정 프로바이더에 묶이면 다운타임 시 사회적 능력 손실. failover·redundancy 설계 없이는 확장 금물.
