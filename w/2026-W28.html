🧪 기회 — 무엇을 만들까

  • custom harness용 edit tool을 모델별 training에 맞춰 여러 변형으로 구현하고 자동 선택하는 adapter — 'better models, worse tools' 회귀를 흡수
  • spec + eval suite에 대해 반복 실행하는 minimal loop harness(재시작·수렴·over-edit 로깅)로 software factory 패턴을 실전 검증
  • coding agent의 주간 토큰 지출을 실시간 감지·상한하는 tokenomics 대시보드 — Harrison Chase가 지적한 '조용히 수천 달러' 문제 대응
  • local/hybrid 배포용 full system(search·tools·harness·ingestion)을 묶은 sovereign AI 스택 — frontier 격차 축소 활용
  • cross-model review 파이프라인(한 벤더 모델이 다른 벤더 산출물 리뷰)으로 P1 버그 검출률 향상
  • agent끼리 학습을 공유하는 'Stack Overflow for agents' 및 portable spec-driven skill 패키징

⚠️ 리스크 — 무엇을 경계할까

  • Sonnet 5의 headline 가격에 현혹되지 말 것 — tokenizer 변경으로 turn당 토큰·cost-per-task가 실제로는 오를 수 있음(영어 ~1.4x)
  • 'better model = better tool use'라는 가정 금물 — SOTA 모델이 custom schema에서 필드를 지어내 tool call을 깨뜨림
  • autonomous software factory 하이프를 곧이곧대로 받지 말 것 — engineering discipline이 hype에 뒤처졌는지 loop 수렴을 직접 검증
  • 경쟁 벤더의 AI code-review agent끼리 무한 disagreement loop에 빠져 수만 달러 inference 비용 발생 가능
  • cognitive debt — agent가 대규모 변경을 만들수록 개발자의 코드 이해가 실제 동작에서 멀어짐; 프로덕션 리뷰를 멈추지 말 것
  • prompt injection이 되돌릴 수 없는 피해를 낼 수 있는 곳에는 agent 배치 금지; 6,000회 실패한 공격은 보안 보장이 아님
  • unpinned dependency의 supply chain attack — 실험용 tooling만으로도 credential exfiltration 위험
